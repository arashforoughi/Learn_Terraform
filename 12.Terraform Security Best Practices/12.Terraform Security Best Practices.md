# 12. Terraform Security Best Practices

Security is a crucial aspect of Terraform. Poorly managed **secrets, state files, and permissions** can lead to security vulnerabilities. In this chapter, we'll cover how to **protect sensitive data, secure Terraform state, and enforce role-based access controls (RBAC).**

## **12.1 Managing Secrets in Terraform**

Terraform does not have built-in **secret management**. By default, sensitive values like API keys or passwords may appear in state files or logs. Here are **best practices** to handle secrets safely.

### **🔹 Avoid Hardcoding Secrets** (❌ Bad Practice)

```hcl
provider "aws" {
  access_key = "AKIAEXAMPLE"
  secret_key = "SECRET12345"
}
```

❌ **Problem:** Secrets are stored in plaintext, leading to **security risks**.

### **🔹 Use Environment Variables** (✅ Best Practice)

Instead of hardcoding credentials, store them as **environment variables**.

📌 **Step 1: Set Environment Variables**

```bash
$ export AWS_ACCESS_KEY_ID="AKIAEXAMPLE"
$ export AWS_SECRET_ACCESS_KEY="SECRET12345"
```

📌 **Step 2: Reference Them in Terraform**

```hcl
provider "aws" {
  region = "us-east-1"
}
```

✅ Terraform **automatically** picks up credentials from environment variables.

### **🔹 Store Secrets in AWS Systems Manager Parameter Store (SSM)**

📌 **Step 1: Store a Secret in SSM Parameter Store**

```bash
$ aws ssm put-parameter --name "db_password" --value "SuperSecret123" --type "SecureString"
```

📌 **Step 2: Retrieve It in Terraform**

```hcl
data "aws_ssm_parameter" "db_password" {
  name = "db_password"
}

resource "aws_instance" "example" {
  tags = {
    Password = data.aws_ssm_parameter.db_password.value
  }
}
```

✅ Terraform fetches the **securely stored password** from SSM.

### **🔹 Use HashiCorp Vault for Secrets Management**

Vault is an advanced tool for **storing and dynamically generating secrets**.

📌 **Step 1: Store a Secret in Vault**

```bash
$ vault kv put secret/aws access_key="AKIAEXAMPLE" secret_key="SECRET12345"
```

📌 **Step 2: Retrieve It in Terraform**

```hcl
data "vault_generic_secret" "aws" {
  path = "secret/aws"
}

provider "aws" {
  access_key = data.vault_generic_secret.aws.data["access_key"]
  secret_key = data.vault_generic_secret.aws.data["secret_key"]
}
```

✅ Vault keeps secrets **encrypted and securely accessible**.

## **12.2 Securing Terraform State Files**

Terraform **stores state files** that contain **sensitive information** like passwords, API keys, and infrastructure details. **Securing these files** is essential.

### **🔹 Store State Remotely in AWS S3 with Encryption**

Using local state files (`terraform.tfstate`) is **risky**. Instead, store Terraform state in **AWS S3 with encryption**.

📌 **Step 1: Create an S3 Bucket for Terraform State**

```bash
$ aws s3 mb s3://my-terraform-state
$ aws s3api put-bucket-versioning --bucket my-terraform-state --versioning-configuration Status=Enabled
```

📌 **Step 2: Enable State Locking with DynamoDB**

```bash
$ aws dynamodb create-table \
    --table-name terraform-lock \
    --attribute-definitions AttributeName=LockID,AttributeType=S \
    --key-schema AttributeName=LockID,KeyType=HASH \
    --provisioned-throughput ReadCapacityUnits=1,WriteCapacityUnits=1
```

📌 **Step 3: Configure Terraform Backend**

```hcl
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-lock"
  }
}
```

✅ This **secures state files in S3 with encryption** and enables **state locking** to prevent conflicts.

### **🔹 Restrict State File Access Using IAM Policies**

📌 **IAM Policy to Restrict Access to Terraform State in S3**

```json
{
    "Effect": "Deny",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::my-terraform-state/terraform.tfstate",
    "Condition": {
        "StringNotEqualsIfExists": {
            "aws:username": "terraform-user"
        }
    }
}
```

✅ This **prevents unauthorized users** from accessing Terraform state.

## **12.3 Role-Based Access Control (RBAC) in Terraform**

Using **IAM roles** and **RBAC policies** ensures that only authorized users can make Terraform changes.

### **🔹 Use IAM Roles Instead of Static Credentials**

📌 **Step 1: Create an IAM Role for Terraform**

```bash
$ aws iam create-role --role-name TerraformRole --assume-role-policy-document file://trust-policy.json
```

📌 **Step 2: Assign the Role in Terraform**

```hcl
provider "aws" {
  region = "us-east-1"
  assume_role {
    role_arn = "arn:aws:iam::123456789012:role/TerraformRole"
  }
}
```

✅ Terraform now **uses temporary credentials** instead of storing secrets.

### **🔹 Restrict Terraform Operations by Role**

📌 **Example: Allow Terraform to Modify Only EC2 Instances**

```json
{
    "Effect": "Allow",
    "Action": [
        "ec2:RunInstances",
        "ec2:TerminateInstances",
        "ec2:DescribeInstances"
    ],
    "Resource": "*"
}
```

✅ This policy ensures Terraform **cannot modify other resources** (e.g., S3, RDS).

## **Scenario: Store AWS Credentials Securely Using Terraform Environment Variables**

### **❌ Problem: Hardcoded Credentials in Terraform**

```hcl
provider "aws" {
  access_key = "AKIAEXAMPLE"
  secret_key = "SECRET12345"
}
```

❌ This exposes **sensitive information**.

### **✅ Solution: Use Environment Variables**

📌 **Step 1: Store AWS Credentials Securely**

```bash
$ export AWS_ACCESS_KEY_ID="AKIAEXAMPLE"
$ export AWS_SECRET_ACCESS_KEY="SECRET12345"
```

📌 **Step 2: Use Default Authentication in Terraform**

```hcl
provider "aws" {
  region = "us-east-1"
}
```

✅ Terraform automatically **retrieves credentials from environment variables**.
