# 13. Integrating Terraform with CI/CD Pipelines

Integrating **Terraform with CI/CD** ensures automated, consistent, and reliable infrastructure deployment. This chapter covers **GitLab CI/CD integration, GitOps workflows, and automation strategies**.

## **13.1 Using Terraform with GitLab CI/CD**

Terraform can be integrated into **GitLab CI/CD** to **automate infrastructure provisioning**.

### **🔹 Step 1: Create a GitLab Repository for Terraform**

- Initialize a **new Git repository**
- Add Terraform configuration files (`main.tf`, `variables.tf`, etc.)

```bash
$ git init
$ git add .
$ git commit -m "Initial commit"
$ git remote add origin <your-gitlab-repo>
$ git push -u origin main
```

### **🔹 Step 2: Create a GitLab CI/CD Pipeline for Terraform**

📌 **GitLab uses `.gitlab-ci.yml` to define CI/CD pipelines**.

```yaml
stages:
  - validate
  - plan
  - apply

variables:
  TF_VERSION: "1.6.0"
  AWS_REGION: "us-east-1"

before_script:
  - apt-get update && apt-get install -y unzip
  - curl -o terraform.zip https://releases.hashicorp.com/terraform/${TF_VERSION}/terraform_${TF_VERSION}_linux_amd64.zip
  - unzip terraform.zip && mv terraform /usr/local/bin/
  - terraform --version

terraform_validate:
  stage: validate
  script:
    - terraform init
    - terraform fmt -check
    - terraform validate

terraform_plan:
  stage: plan
  script:
    - terraform init
    - terraform plan -out=tfplan
  artifacts:
    paths:
      - tfplan

terraform_apply:
  stage: apply
  when: manual
  script:
    - terraform apply -auto-approve tfplan
```

✅ **What This Does:**

- **Installs Terraform**
- **Validates configuration**
- **Plans infrastructure changes**
- **Applies Terraform changes manually** (to prevent accidental updates)

### **🔹 Step 3: Secure Terraform State Using S3 Backend**

Modify `main.tf` to use a **remote backend** (e.g., AWS S3):

```hcl
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "gitlab/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-lock"
  }
}
```

✅ **Terraform state is securely stored and locked via DynamoDB.**

### **🔹 Step 4: Store AWS Credentials in GitLab CI/CD Variables**

Instead of hardcoding credentials, store them in **GitLab CI/CD → Settings → CI/CD → Variables**:

| Key | Value | Type |
| --- | --- | --- |
| AWS_ACCESS_KEY_ID | AKIAEXAMPLE | Masked |
| AWS_SECRET_ACCESS_KEY | SECRET12345 | Masked |

Then, update `.gitlab-ci.yml`:

```yaml
variables:
  AWS_ACCESS_KEY_ID: $AWS_ACCESS_KEY_ID
  AWS_SECRET_ACCESS_KEY: $AWS_SECRET_ACCESS_KEY
```

✅ **Sensitive credentials are securely managed.**

## **13.2 Terraform GitOps Workflow**

**GitOps** is a declarative approach where **infrastructure changes are managed via Git**.

### **🔹 Step 1: Implement a Pull Request Workflow**

- Developers submit **Merge Requests (MRs)** for Terraform changes.
- The pipeline runs `terraform plan`.
- If approved, `terraform apply` is triggered.

✅ **Benefits:**

- **Version-controlled infrastructure**
- **Rollback capabilities**
- **Collaborative approval process**

### **🔹 Step 2: Enforce Terraform Formatting & Linting**

Modify `.gitlab-ci.yml` to enforce Terraform best practices:

```yaml
terraform_lint:
  stage: validate
  script:
    - terraform fmt -check
    - terraform validate
```

✅ This prevents **misformatted** or **invalid configurations** from being deployed.

## **13.3 Automating Infrastructure Deployment**

Using GitLab CI/CD, **Terraform changes can be automatically applied** in different environments.

### **🔹 Step 1: Define Multiple Environments**

```yaml
stages:
  - validate
  - plan
  - apply

deploy_dev:
  stage: apply
  environment:
    name: dev
  script:
    - terraform apply -auto-approve
  only:
    - dev

deploy_prod:
  stage: apply
  environment:
    name: production
  script:
    - terraform apply -auto-approve
  only:
    - main
```

✅ **Changes in `dev` branch deploy to the development environment**.

✅ **Changes in `main` branch deploy to production**.

### **🔹 Step 2: Protect the `main` Branch**

- **Require approvals before applying changes**
- **Use Merge Requests (MRs) for all updates**

✅ Prevents **accidental production changes**.
