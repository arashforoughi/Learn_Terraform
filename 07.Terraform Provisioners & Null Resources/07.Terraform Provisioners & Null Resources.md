# 7. Terraform Provisioners & Null Resources

Terraform **Provisioners** allow us to execute scripts and commands **on local or remote machines** after creating infrastructure. They help with **bootstrapping**, configuration management, and setting up dependencies.

## **7.1 When to Use Provisioners?**

Provisioners should be used **only when necessary** because Terraform is designed to **manage infrastructure, not configuration**.

🔹 **Use provisioners when:**

✅ Installing software (e.g., `nginx` on an EC2 instance)

✅ Copying files to remote servers

✅ Running one-time scripts for setup

🔹 **Avoid provisioners when:**

❌ Config management tools (e.g., Ansible, Puppet) are available

❌ Cloud-init or user-data can achieve the same results

## **7.2 Types of Provisioners**

Terraform supports three main provisioners:

1️⃣ **local-exec** – Runs commands on the local machine

2️⃣ **remote-exec** – Runs commands on the remote instance

3️⃣ **file** – Copies files from the local machine to the remote instance

## **7.3 Using `local-exec` Provisioner**

📌 The `local-exec` provisioner runs a script **on the machine running Terraform**.

🔹 **Example: Notify via Slack after EC2 instance creation**

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  provisioner "local-exec" {
    command = "echo 'EC2 instance ${self.id} has been created' >> ec2-log.txt"
  }
}
```

➡ Terraform **logs the instance ID** into `ec2-log.txt` after creating the instance.

## **7.4 Using `remote-exec` Provisioner**

📌 The `remote-exec` provisioner runs commands **inside the remote server**.

🔹 **Example: Install Nginx on EC2 after Terraform creates it**

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  key_name      = "my-key"   # Ensure SSH access
  security_groups = ["allow_ssh_http"]

  connection {
    type        = "ssh"
    user        = "ubuntu"
    private_key = file("~/.ssh/my-key.pem")
    host        = self.public_ip
  }

  provisioner "remote-exec" {
    inline = [
      "sudo apt update -y",
      "sudo apt install nginx -y",
      "sudo systemctl start nginx"
    ]
  }
}
```

➡ After creating the EC2 instance, Terraform logs in via SSH and **installs Nginx**.

## **7.5 Using `file` Provisioner**

📌 The `file` provisioner copies files **from the local machine to the remote server**.

🔹 **Example: Copy an application config file to EC2**

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  key_name      = "my-key"

  connection {
    type        = "ssh"
    user        = "ubuntu"
    private_key = file("~/.ssh/my-key.pem")
    host        = self.public_ip
  }

  provisioner "file" {
    source      = "app-config.conf"
    destination = "/etc/nginx/conf.d/app-config.conf"
  }
}
```

➡ Terraform **copies `app-config.conf` to `/etc/nginx/conf.d/` on the EC2 instance**.

## **7.6 Using Null Resources**

📌 **What is a Null Resource?**

A **null resource** is a Terraform resource that **does nothing** except trigger provisioners when dependencies change.

🔹 **Example: Run a script every time a VPC changes**

```hcl
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}

resource "null_resource" "vpc_change" {
  triggers = {
    vpc_id = aws_vpc.main.id
  }

  provisioner "local-exec" {
    command = "echo 'VPC changed!'"
  }
}
```

➡ If the VPC changes, Terraform **runs the local script again**.

## **7.7 Avoiding Anti-Patterns with Provisioners**

🔴 **Anti-Pattern 1:** Using provisioners for configuration management

✅ Instead, use **Ansible, Chef, or Puppet** for ongoing config updates.

🔴 **Anti-Pattern 2:** Hardcoding sensitive credentials in scripts

✅ Instead, store **secrets in Vault or AWS Secrets Manager**.

🔴 **Anti-Pattern 3:** Running provisioners every time Terraform applies

✅ Instead, use **triggers in `null_resource`** to avoid unnecessary execution.

## **Example: Run a Shell Script on EC2 After Creation**

📌 This example **copies a script to EC2** and runs it after creation.

📁 **Project Structure**

```bash
terraform-project/
│── main.tf                 # Terraform config
│── setup.sh                # Script to run on EC2
```

📌 **Create `setup.sh` (Local Machine)**

```bash
#!/bin/bash
sudo apt update -y
sudo apt install -y apache2
sudo systemctl start apache2
sudo systemctl enable apache2
```

📌 **Define Terraform Configuration (`main.tf`)**

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
	count         = "20"
  key_name      = "my-key"

  connection {
    type        = "ssh"
    user        = "ubuntu"
    private_key = file("~/.ssh/my-key.pem")
    host        = self.public_ip
  }

  provisioner "file" {
    source      = "setup.sh"
    destination = "/home/ubuntu/setup.sh"
  }

  provisioner "remote-exec" {
    inline = [
      "chmod +x /home/ubuntu/setup.sh",
      "sudo /home/ubuntu/setup.sh"
    ]
  }
}
```

📌 **Apply Terraform**

```bash
$ terraform init
$ terraform apply -auto-approve
```

➡ Terraform **creates an EC2 instance, copies `setup.sh`, and runs it**.
