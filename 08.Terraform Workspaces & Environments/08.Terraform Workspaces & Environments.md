# 8. Terraform Workspaces & Environments

Terraform **workspaces** provide an easy way to manage multiple environments (e.g., **dev, staging, prod**) within a single Terraform configuration. Instead of maintaining separate Terraform directories, you can use **workspaces** to dynamically switch between environments.

## **8.1 What Are Workspaces?**

A **Terraform workspace** is an **isolated state environment** within the same Terraform project.

🔹 By default, Terraform has a **default workspace**

🔹 Additional workspaces can be created and switched dynamically

🔹 Each workspace has its own **state file**, preventing conflicts

✅ **Why Use Workspaces?**

✔ **Simplifies multi-environment deployments**

✔ **Reduces duplication** (no need for separate Terraform projects)

✔ **Keeps state files isolated for different environments**

🚨 **When NOT to Use Workspaces?**

❌ When deploying to different **AWS accounts** – use different backend configurations instead

❌ When using Terraform for very large environments – separate Terraform projects may be better

## **8.2 Using Workspaces for Multiple Environments**

### **🔹 Step 1: Check the Current Workspace**

```bash
terraform workspace show
```

➡ This shows the **current active workspace** (default if none is created).

### **🔹 Step 2: Create a New Workspace**

```bash
$ terraform workspace new dev
$ terraform workspace new staging
$ terraform workspace new prod
```

➡ This creates three workspaces: **dev, staging, and prod**.

### **🔹 Step 3: List Available Workspaces**

```bash
$ terraform workspace list
```

➡ The output will look like:

```
  default
* dev
  staging
  prod
```

(The **`*`** indicates the active workspace.)

### **🔹 Step 4: Switch Between Workspaces**

```bash
$  terraform workspace select staging
```

➡ This switches to the **staging** workspace.

### **🔹 Step 5: Use Workspaces in Terraform Configuration**

Terraform provides a special built-in variable:

```hcl
terraform.workspace
```

You can use this variable to **dynamically adjust resources** based on the active workspace.

📌 **Example: Use Different AWS Tags for Each Workspace**

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  tags = {
    Name        = "web-${terraform.workspace}"
    Environment = terraform.workspace
  }
}
```

➡ If you apply Terraform in the **staging workspace**, the instance name will be **web-staging**.

### **🔹 Step 6: Define Variables Based on Workspaces**

📌 Instead of using a single static value, we can use **workspace-specific configurations**.

📁 **Create `variables.tf`**

```hcl
variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t2.micro"
}
```

📌 **Modify `main.tf` to Use Workspaces**

```hcl
locals {
  instance_type_map = {
    "default" = "t2.micro"
    "dev"     = "t3.micro"
    "staging" = "t3.medium"
    "prod"    = "t3.large"
  }
}

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = lookup(local.instance_type_map, terraform.workspace, "t2.micro")

  tags = {
    Name        = "web-${terraform.workspace}"
    Environment = terraform.workspace
  }
}
```

➡ When switching workspaces, Terraform dynamically **assigns different instance types**.

## **8.3 Best Practices for Managing Dev, Staging, and Production**

🔹 **Use Workspaces for Small Environments** – If your environments are simple, workspaces are great.

🔹 **Use Separate Backends for Different AWS Accounts** – Workspaces are NOT meant for managing separate AWS accounts.

🔹 **Store State Remotely** – Use an **S3 backend** to avoid losing workspace states.

🔹 **Use CI/CD to Deploy Workspaces** – Automate workspace selection in GitLab, Jenkins, or GitHub Actions.

## **Scenario: Create Different Environments Using Workspaces**

📌 **Goal:** Deploy EC2 instances in **dev, staging, and prod** using Terraform workspaces.

📁 **Project Structure**

```bash
terraform-workspaces/
│── main.tf                   # Terraform config
│── variables.tf              # Variables file
│── backend.tf                # Remote state storage
│── outputs.tf                # Output file
```

📌 **`backend.tf` (Store State in S3)**

```hcl
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "terraform-workspaces/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-lock"
  }
}
```

➡ This ensures **each workspace has its own isolated state** in S3.

📌 **`main.tf` (Define EC2 Instance per Workspace)**

```hcl
provider "aws" {
  region = "us-east-1"
}

locals {
  instance_type_map = {
    "default" = "t2.micro"
    "dev"     = "t3.micro"
    "staging" = "t3.medium"
    "prod"    = "t3.large"
  }
}

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = lookup(local.instance_type_map, terraform.workspace, "t2.micro")

  tags = {
    Name        = "web-${terraform.workspace}"
    Environment = terraform.workspace
  }
}
```

📌 **Switch Between Environments & Deploy**

```bash
$ terraform init

$ terraform workspace select dev
$ terraform apply -auto-approve

$ terraform workspace select staging
$ terraform apply -auto-approve

$ terraform workspace select prod
$ terraform apply -auto-approve
```

➡ Terraform **creates different instances for each environment**, tagged accordingly.
