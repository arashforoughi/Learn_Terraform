# 4. Terraform State Management

Terraform uses a **state file** (`terraform.tfstate`) to keep track of the infrastructure it manages. In this chapter, we'll dive into **how Terraform state works, local vs. remote state, and best practices for securing and locking state**.

## **4.1 What is Terraform State?**

Terraform maintains the **state** of the infrastructure to:
✔ Track resource dependencies

✔ Map Terraform resources to real-world infrastructure

✔ Improve performance by caching data

✔ Enable collaboration when using remote state

📌 **Example: Terraform State File (`terraform.tfstate`)**
After running `terraform apply`, Terraform generates `terraform.tfstate`, which contains:

```json
{
  "resources": [
    {
      "type": "aws_s3_bucket",
      "name": "my_bucket",
      "instances": [
        {
          "attributes": {
            "bucket": "my-terraform-bucket-123",
            "arn": "arn:aws:s3:::my-terraform-bucket-123"
          }
        }
      ]
    }
  ]
}
```

➡ This file **maps** Terraform resources to actual AWS resources.

## **4.2 Local vs. Remote State**

By default, Terraform stores the state **locally** in `terraform.tfstate`.

For **team collaboration**, it's best to use a **remote backend** (e.g., AWS S3, Terraform Cloud).

### **🔹 Local State (Default)**

✔ Easy to use

❌ Hard to share

❌ Risk of accidental deletion

```bash
$ terraform apply
$ cat terraform.tfstate
```

➡ Stores state locally in `terraform.tfstate`.

### **🔹 Remote State (Best for Teams)**

✔ Shared access

✔ Prevents conflicts

✔ Secure storage

❌ Requires extra setup

Example **backends**:

- **AWS S3**
- **Azure Blob Storage**
- **Google Cloud Storage (GCS)**
- **Terraform Cloud**

## **4.3 Terraform State Locking & Security**

When multiple users modify infrastructure at the same time, it can cause **state conflicts**.

🔹 **State Locking** prevents concurrent modifications by locking the state file.

🔹 **DynamoDB** is used for **state locking** when storing state in AWS S3.

## **4.4 Managing State with S3 + DynamoDB for Locking**

We'll configure **Terraform remote state storage** in **AWS S3** and **enable locking** with **DynamoDB**.

### **🔹 Step 1: Create an S3 Bucket for State Storage**

Run the following command to create an S3 bucket:

```bash
$ aws s3api create-bucket --bucket my-terraform-state-bucket --region us-east-1
```

📌 Enable versioning to protect the state file:

```bash
$ aws s3api put-bucket-versioning --bucket my-terraform-state-bucket --versioning-configuration Status=Enabled
```

### **🔹 Step 2: Create a DynamoDB Table for Locking**

DynamoDB prevents **multiple users from modifying the state at the same time**.

```bash
$ aws dynamodb create-table --table-name terraform-lock \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

📌 **LockID** is a unique key for state locks.

### **🔹 Step 3: Configure Terraform to Use Remote State**

Modify `backend.tf` to store Terraform state in S3:

```hcl
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-lock"
    encrypt        = true
  }
}
```

### **🔹 Step 4: Initialize and Apply Terraform**

After setting up the backend, initialize Terraform:

```bash
$ terraform init
```

➡ Terraform moves the **local state** to **S3**.

Run `terraform apply` to test:

```bash
$ terraform apply -auto-approve
```

➡ Terraform now **locks the state file** in DynamoDB when applying changes.

## **4.5 Verifying State Storage and Locking**

### **🔹 Check State File in S3**

Run:

```bash
$ aws s3 ls s3://my-terraform-state-bucket
```

➡ You should see `terraform.tfstate`.

### **🔹 Check State Lock in DynamoDB**

Run:

```bash
$ aws dynamodb scan --table-name terraform-lock
```

➡ Shows active locks if any.
