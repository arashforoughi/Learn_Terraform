# 9. Managing Terraform Backends

Terraform **backends** define **where and how Terraform stores its state files**. By default, Terraform stores state **locally** in `terraform.tfstate`, but **remote backends** (like AWS S3, Azure Blob, or Google Cloud Storage) provide a better way to **share and secure** state files.

## **9.1 Local Backend vs. Remote Backend**

| Feature | Local Backend | Remote Backend (S3, Azure Blob, GCS) |
| --- | --- | --- |
| **Storage Location** | Locally on the machine (`terraform.tfstate`) | Cloud storage (S3, Azure Blob, GCS, Terraform Cloud) |
| **Collaboration** | Hard to share | Multiple users can access |
| **Security** | Risk of loss | Cloud security, encryption, IAM control |
| **State Locking** | ❌ No state locking | ✅ Supports state locking (e.g., DynamoDB for S3) |
| **Versioning** | ❌ No automatic versioning | ✅ Cloud storage versioning |

## **9.2 Configuring Terraform Backends**

### **🔹 Local Backend (Default)**

If no backend is configured, Terraform stores state in the project directory:

📁 **Example: Default Local Backend**

```
terraform/
│── main.tf
│── terraform.tfstate
```

➡ **Problems with Local Backend:**

❌ Difficult to collaborate with multiple team members

❌ No state locking → risk of corrupting state

❌ If the local machine crashes, state is lost

## **9.3 Storing Terraform State in Remote Backends**

### **🔹 Using AWS S3 as a Remote Backend**

A **remote backend** (like AWS S3) allows teams to **store and access** Terraform state **securely**.

📌 **Step 1: Create an S3 Bucket for State Storage**

```bash
$ aws s3api create-bucket --bucket my-terraform-state-bucket --region us-east-1
```

📌 **Step 2: Enable Versioning on the S3 Bucket**

```bash
$ aws s3api put-bucket-versioning --bucket my-terraform-state-bucket --versioning-configuration Status=Enabled
```

📌 **Step 3: Configure `backend.tf` for S3**

```hcl
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
  }
}
```

➡ When `terraform init` is run, Terraform stores the state in S3.

### **🔹 Using Azure Blob Storage as a Remote Backend**

📌 **Step 1: Create an Azure Storage Account & Container**

```
$ az storage account create --name terraformstate123 --resource-group myResourceGroup --location eastus --sku Standard_LRS
$ az storage container create --name terraformstate --account-name terraformstate123
```

📌 **Step 2: Configure `backend.tf` for Azure**

```hcl
terraform {
  backend "azurerm" {
    resource_group_name   = "myResourceGroup"
    storage_account_name  = "terraformstate123"
    container_name        = "terraformstate"
    key                   = "terraform.tfstate"
  }
}
```

➡ Running `terraform init` will configure the remote backend.

### **🔹 Using Google Cloud Storage (GCS) as a Remote Backend**

📌 **Step 1: Create a GCS Bucket**

```bash
$ gsutil mb -l us-east1 gs://my-terraform-state-bucket/
```

📌 **Step 2: Configure `backend.tf` for GCS**

```hcl
terraform {
  backend "gcs" {
    bucket  = "my-terraform-state-bucket"
    prefix  = "terraform/state"
  }
}
```

➡ Running `terraform init` will store the state in Google Cloud Storage.

## **9.4 State Locking with DynamoDB for AWS S3**

If multiple users apply Terraform at the same time, it can cause state corruption. **State locking** prevents this by allowing only one user to modify state at a time.

### **🔹 Step 1: Create a DynamoDB Table for Locking**

```bash
$ aws dynamodb create-table \
  --table-name terraform-lock \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

### **🔹 Step 2: Modify `backend.tf` to Enable Locking**

```hcl
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-lock"
  }
}
```

➡ Now, Terraform will **automatically lock the state file** when in use.

## **9.5 Scenario: Migrate Terraform State from Local to S3 Backend with State Locking**

### **🔹 Step 1: Initialize Local Terraform**

```bash
$ terraform init
$ terraform apply -auto-approve
```

➡ This creates infrastructure and saves the state **locally**.

### **🔹 Step 2: Add Remote Backend (`backend.tf`)**

```hcl
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-lock"
  }
}
```

### **🔹 Step 3: Migrate the State to S3**

```bash
$ terraform init -migrate-state
```

➡ Terraform **detects the change** and migrates state from local to S3.

### **🔹 Step 4: Verify the Remote State**

```bash
$ aws s3 ls s3://my-terraform-state-bucket
```

➡ You should see the `terraform.tfstate` file.

### **🔹 Step 5: Apply Configuration Using Remote State**

```bash
$ terraform apply -auto-approve
```

➡ Terraform now **stores and locks state remotely** in S3.
